# 構造定義およびリスク評価報告

提示された「切り抜き字幕作成ツール MVP」の仕様、リポジトリ構成、およびAI向け引き継ぎ仕様を網羅的に分析した。
本システムはローカル完結型の字幕・カット編集前処理ツールとして非常に強固な構造を持つが、実装および配布フェーズにおいて検知された潜在的リスク、ならびに構造的脆弱性を以下に列挙する。

## 1. 致命的リスク: GPLv3ライセンス汚染および配布構造の脆弱性

本リポジトリは GPL-3.0-or-later での配布を前提としているが、同梱または自動取得される外部バイナリとのライセンス競合リスクが存在する。

### FFmpeg (BtbNビルド) のライセンス競合

BtbNが配布するFFmpegビルドには、GPL版とLGPL版、さらに非自由 (Non-free) なコーデック (libfdk-aac等) を含むものが存在する。非自由コーデックを含むバイナリをGPLv3の成果物に同梱、あるいは自動スクリプト (`download-ffmpeg.bat`) で未検証のまま結合・配布した場合、ライセンス違反による法的リスクが発生する。

### whisper.cpp (MITライセンス) との結合

whisper.cpp 自体は MIT ライセンスであり、GPLv3との互換性はあるが、ビルド環境 (Vulkan/AMD SDK等) にプロプライエタリなライブラリが動的・静的にリンクされている場合、配布時にライセンスの伝播範囲を巡る法解釈の脆弱性が生じる。

## 2. アーキテクチャおよびシステム構造の脆弱性

### ポート競合とプロセス管理の不完全性

`launch.bat` 起動時に既存サーバーの `/api/version` を検知して旧ビルドを停止する仕様となっているが、ポートが別プロセスに占有されていた場合、またはゾンビプロセス化した Uvicorn がシグナルを正常に受信しない場合、ポートバインドエラー (EADDRINUSE) により起動が沈黙する。

### ステート (`edit_plan.json`) の破壊リスク

システムの中心データが単一の `edit_plan.json` に依存している。UIでの編集操作、またはCLI (`run-pipeline`) による外部オーケストレーションからの書き込み処理中に、例外 (ディスクフル、強制終了) が発生した場合、ファイル破損に対するフォールバック (世代管理・ジャーナリング) が明示されていない。これはデータロストの単一障害点 (SPOF) である。

### I/Oボトルネックと一時ファイルの肥大化

プロジェクト毎に `audio/source_range.wav` や `preview/preview_low.mp4`、さらに `analysis/waveform.json` などの重い中間ファイルを `projects/<project_id>/` 内部に生成する設計であるため、長尺動画の一括処理時 (CLIモード) にディスクI/Oが急増し、システム全体のパフォーマンスが著しく低下する。

## 3. レッドチーム (仮想敵) 視点による攻撃・エラーシナリオ

```
[動画入力] ──> [FFmpeg 音声抽出] ──> [whisper.cpp (Vulkan)] ──> [edit_plan.json] ──> [FFmpeg 焼き込み]
                      │                           │                      │
                 (VFRによる)                (VRAM枯渇による)        (不正な文字・改行)
                [音ズレ発生]               [プロセス強制終了]       [SRTパースエラー]
```

### 可変フレームレート (VFR) 動画による音ズレ波及

スマホや画面キャプチャ由来のVFR動画が入力された場合、FFmpegでの音声抽出時 (CFR変換なし) にタイムスタンプのズレが生じる。結果として、Whisperが正しく文字起こしを行っても、`edit_plan.json` を経由した最終出力 (`final.mp4`) で字幕と映像が致命的に乖離する。

### whisper.cpp (Vulkan) のVRAM管理不足

GPUプロファイル (`compute_profile=gpu`) 実行時、高解像度動画の処理や複数割込みによってVRAMが枯渇 (OOM) した場合、`whisper-cli.exe` は例外を返さずに即座にクラッシュする可能性がある。この際、監査ログ (`processing.jsonl`) には不完全なデータ、またはタイムアウトのみが記録され、原因特定が困難になる。

### SRT焼き込み時の特殊文字・エスケープ制限

ユーザーがブラウザで字幕を手動編集した際、SRTの規格を破る改行コード、制御文字、またはFFmpegの `subtitles` フィルターが誤認する特殊文字 (`[`, `]`, `:`, `\` 等) が混入した場合、最終エクスポート処理 (`export`) 全体が異常終了する。

## 4. 推奨される具体的防衛策

### FFmpeg取得先の厳密化

`download-ffmpeg.bat` が取得するバイナリを「GPL準拠かつNon-freeコード不含」のものに固定し、ハッシュ値 (SHA-256) 検証をスクリプト内に強制挿入する。

### アトミックなファイル書き込みの導入

`edit_plan.json` を更新する際は、一度 `edit_plan.json.tmp` に書き込み、正常完了後にリネーム (置換) する処理構造へ変更し、SPOFを排除する。

### 音声抽出時の強制CFR化

`extract-audio` フェーズにおいて、FFmpegに `-vsync cfr` (または最新の `-fps_mode cfr`) 引数を強制し、時間軸の同期精度を担保する。

## 5. まとめ

このシステムは、ローカル完結・非破壊・編集中心という構造上の利点を持つ一方、配布バイナリのライセンス管理、単一ファイルへの状態依存、VFR動画処理、GPU OOM、SRTエスケープといった点に明確なリスクがある。
配布・外部連携を前提にするなら、導入物の検証、アトミック更新、同期精度の補強を優先して実装すべきである。
