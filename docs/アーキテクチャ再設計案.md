# 切り抜き動画工房 アーキテクチャ再設計案

この文書は、現状の「CLI 部品を GUI がラップする」構成を、破綻しにくい形に整理し直すための設計方針をまとめたものです。

## 1. 問題認識

現在の実装は、字幕、無音検出、VAD、話者分離、枠、テキスト装飾、画面エフェクト、最終出力が増えた結果、次の問題を起こしやすくなっています。

- GUI と CLI が同じ設定を別々に持ちやすい
- 1 機能を足すたびに export 経路へ継ぎ足しが必要になる
- プレビューでは見えるが最終出力に反映されない差分が出やすい
- 設定の保存先が分散して、どれが正か分かりにくい

## 2. 再設計の原則

### 原則 A: 単一の正本を持つ

正本は `project.json` と `edit_plan.json` に寄せる。

- `project.json`
  - プロジェクトの基本情報
  - GUI の表示状態
  - 出力プロファイル
  - 保存済みプリセット参照
- `edit_plan.json`
  - どこを残すか
  - どこを削るか
  - 字幕の時刻変換
  - 話者ラベル
  - シーン割り当て
  - 手動修正結果

GUI の内部 state は一時的な編集バッファに留め、保存時に正本へ反映する。

### 原則 B: コアは UI を知らない

字幕抽出、無音検出、VAD、話者分離、字幕補正、カット計算、ASS 生成、最終書き出しはすべて `core` 層に置く。

GUI は次だけを担当する。

- プロジェクト選択
- 設定編集
- プレビュー表示
- 実行ボタン
- 結果の再読み込み

### 原則 C: CLI は部品単位の実行器にする

CLI は「独自ロジック」ではなく「コア関数の呼び出し口」にする。

例:

- `probe`
- `new-project`
- `extract-audio`
- `transcribe`
- `detect-silence`
- `detect-vad`
- `build-edit-plan`
- `render-preview`
- `export-final`
- `decorate-preview`
- `decorate-export`
- `cleanup`

これにより、GUI と CLI の差分をコマンド構成ではなく UI の違いだけに抑えられる。

### 原則 D: 失敗時に中間成果物を残す

長尺処理では `auto-cleanup` を既定にしない。

- 成功時のみ自動整理
- 失敗時は中間ファイルを保持
- 監査用ログは必ず残す

## 3. 推奨レイヤー構成

```text
backend/app/
  core/
    audio.py
    transcription.py
    vad.py
    silence.py
    edit_plan.py
    decoration.py
    ffmpeg.py
    export.py
    schemas.py
  adapters/
    cli.py
    api.py
    gui_bridge.py
  services.py
  main.py
  cli.py
frontend/
  app.js
  index.html
```

### core

純粋な処理関数を置く。

- 入力: dict / Path / 文字列 / 数値
- 出力: dict / list / Path
- UI 依存を持たない

### adapters

入力変換と出力整形だけを担当する。

- CLI 引数を core に渡す
- FastAPI の request/response を core に渡す
- GUI の保存操作を core に渡す

### services

現状の共通サービス層。

- ファイル操作
- プロジェクト管理
- FFmpeg 実行
- ログ
- 互換性維持

## 4. ワークフローの整理

工程は長くしない。1 工程は 1 部品、1 保存先、1 成功判定に寄せる。

### 音声・字幕系

1. `extract-audio`
2. `detect-vad` または `detect-silence`
3. `transcribe`
4. `build-edit-plan`
5. `save-subtitles`

### 装飾系

1. `load-decoration`
2. `apply-frame`
3. `apply-text-preset`
4. `apply-screen-effects`
5. `render-decoration-preview`
6. `decorate-export`

### 出力系

1. `export-final`
2. `cleanup`

### 工程管理のルール

- 1 工程で触る core 関数は 1 つを原則にする
- 1 工程で更新する正本 JSON は 1 種類を原則にする
- 失敗したら、その工程だけを再実行できるようにする
- GUI の 1 ボタンに複数の副作用を持たせすぎない
- 不要要素は decoration / preset / optional module 側へ逃がす

## 5. GUI と CLI の役割分離

### GUI

GUI は「何を使うか」を選ぶ。

- どのモデルを使うか
- どの VAD モードを使うか
- どのプリセットを使うか
- どの区間を保護するか
- どの装飾セットを適用するか

### CLI

CLI は「どの処理を実行するか」を固定する。

- 部品ごとに実行できる
- 外部 AI から呼べる
- 再現性を保ちやすい

## 6. 保存戦略

保存対象は次の 4 つに整理する。

1. `project.json`
2. `edit_plan.json`
3. `decoration/decoration_project.json`
4. `presets/*.json`

特にプリセットはプロジェクト個別と共通の 2 系統に分ける。

- 共通: 再利用する既定値
- 個別: そのプロジェクトだけの調整値

## 7. 画面反映ルール

最終出力とプレビューの差分を減らすため、次を固定する。

- プレビューは「最終出力の縮小版」
- 装飾プレビューは「装飾 export の縮小版」
- GUI の見た目用 CSS は export ロジックと分離する

つまり、GUI で見えているものは最終出力の概略であり、別実装にしない。

## 8. 具体的な修正優先順位

1. core の入出力スキーマを確定する
2. `edit_plan.json` を唯一の編集結果に固定する
3. CLI から直接呼べる部品コマンドを整理する
4. GUI の各ボタンをコア関数に直結する
5. 最終出力と decoration export の差分を減らす
6. 失敗時のログと中間ファイル保持を標準化する

## 9. 判断基準

新機能を追加する時は、次のどちらかを満たすものだけ採用する。

- 既存の `edit_plan.json` に素直に載る
- 既存の `decoration_project.json` に素直に載る

どちらにも載らない機能は、先にコアのデータ構造を増やす。

工程の分解や修正範囲の狭め方は `docs/部品単位工程管理ルール.md` を基準にする。
