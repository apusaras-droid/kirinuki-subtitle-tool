# OBS映像フィルタープラグインのエフェクト再現技術ドキュメント

## 1. 目的
このドキュメントは、OBS Studio用プラグイン `video_effects_filter` のエフェクト処理を読み解き、動画編集アプリ側で同等の見た目を再現するための設計資料としてまとめたものです。

対象コード:
- [src/video-effects-filter.c](./src/video-effects-filter.c)
- [src/plugin-main.c](./src/plugin-main.c)
- [data/video-effects-filter.effect](./data/video-effects-filter.effect)
- [data/locale/ja-JP.ini](./data/locale/ja-JP.ini)
- [data/locale/en-US.ini](./data/locale/en-US.ini)

## 2. 実装の全体像
このプラグインは、OBSの「映像フィルター」として動作します。実処理はほぼすべてGPUシェーダーで行い、C側は以下の役割だけを担当しています。

- フィルター設定値の保持
- UIプロパティの生成
- 経過時間の加算
- シェーダーへパラメータを送る
- 入力フレームをシェーダーに通して出力する

つまり、動画編集アプリに移植する場合は、Cコード相当の部分は「パラメータ管理」と「フレームごとの時間更新」に置き換えればよく、主要ロジックはシェーダーの再実装が中心になります。

### 2.1 編集対象の階層
動画編集アプリ側では、エフェクト適用対象を少なくとも2階層で分けると扱いやすいです。

- 全体エフェクト
  - 動画全体、またはタイムライン全体に掛ける
  - 例: 全体の色味補正、共通のノイズ、共通の字幕スタイル
- シーンエフェクト
  - 1シーン単位、1コメント単位、1カット単位に掛ける
  - 例: 1発言だけ強調、特定シーンだけ吹き出し枠を変える

この分離をしておくと、「全体の雰囲気」と「個別の演出」を衝突させずに扱えます。

### 2.2 複数エフェクトの束ね方
複数のエフェクトを 1 つのエフェクトとして登録・編集できるようにする場合は、単一のエフェクトIDではなく「エフェクトセット」として扱うのが安全です。

推奨構造:

- `effect`
  - 単体エフェクト
- `effect_group`
  - 単体エフェクトを順序付きで束ねたもの
- `preset`
  - `effect` または `effect_group` に初期値を与えたもの

例:

```json
{
  "id": "emotion_surprise_default",
  "type": "effect_group",
  "name": "驚きプリセット",
  "effects": [
    { "effect_type": "flash", "intensity": 0.8, "speed": 0.6 },
    { "effect_type": "shake", "intensity": 0.4, "speed": 0.7 },
    { "effect_type": "outline", "font_size": 42, "bubble_style": "rounded" }
  ]
}
```

この方式だと、後から各エフェクトを個別に編集しつつ、まとめてプリセット化できます。

## 3. 処理パイプライン
### 3.1 ライフサイクル
1. プラグイン読み込み時に `video_effects_filter` をOBSへ登録する。
2. フィルター作成時に `.effect` ファイルを読み込む。
3. UI変更時に `effect_type` / `intensity` / `speed` / `color` を保存する。
4. 毎フレーム `video_tick` で `time` を加算する。
5. 描画時に解像度、時間、UI値をGPUへ渡してシェーダーを実行する。

### 3.2 シェーダー入力
シェーダーに渡される主なuniformは以下です。

| uniform | 意味 |
| --- | --- |
| `image` | 入力映像テクスチャ |
| `time` | 経過時間。`speed` に比例して増加 |
| `intensity` | 効果の強さ |
| `speed` | アニメーション速度の制御値 |
| `width` | 入力映像の幅 |
| `height` | 入力映像の高さ |
| `color` | 色指定。ネオン、白バランス、パーティクル等で使用 |
| `effect_type` | エフェクト選択ID |

### 3.3 時間の扱い
`video_effects_tick` では、毎フレームの経過秒数に `speed` を掛けて `time` に加算します。  
したがって、`speed = 0` なら時間系の揺れや点滅は止まり、`speed = 1` で標準速度になります。

動画編集アプリで再現する場合は、各クリップのローカル時間またはコンポジション時間を `time` に相当する値として扱えば移植しやすいです。

## 4. UIパラメータ仕様
### 4.1 共通パラメータ
OBS側のUIは共通で3つの主要パラメータを持ちます。

| UI項目 | 型 | 範囲 | 用途 |
| --- | --- | --- | --- |
| `effect_type` | int | 0-61 | 適用するエフェクトID |
| `intensity` | double | 0.0-1.0 | 効果量、ぼかし量、色変化量、しきい値調整など |
| `speed` | double | 0.0-1.0 | 時間依存効果の速度 |
| `color` | RGBA | - | 発光色、補正色、パーティクル色 |

### 4.2 動画編集アプリに置き換える際の考え方
- `intensity` は多くの効果で「ミックス量」として機能する
- `speed` は揺れ、点滅、粒子移動、ランダム変化に使う
- `color` は色指定系の効果にのみ影響する

この設計は、動画編集ソフトの標準パラメータにそのまま落とし込みやすいです。たとえば「Amount」「Rate」「Tint Color」の3系統に割り当てると扱いやすくなります。

### 4.3 シーン用と全体用のUI
UI では、以下の切り替えを用意すると運用しやすいです。

- 適用範囲
  - `全体`
  - `シーン`
  - `コメント単位`
- 適用順
  - `全体 -> シーン -> 個別コメント`
  - 個別コメントが最優先

この順序にしておくと、全体の字幕スタイルとシーン別の強調が競合しにくくなります。

## 5. 画像処理の基本構造
### 5.1 単一シェーダー内の分岐
`.effect` ファイルでは、`effect_type` の値に応じて `if (et == N)` で分岐し、1つのピクセルシェーダーで62種類の効果を処理しています。

これは「エフェクトごとに別プラグインを持つ」のではなく、1本のGPUシェーダーの中で分岐している構成です。

### 5.2 共通的に使われる処理
以下の基礎演算が全体で繰り返し使われています。

- `rand_f()` / `N21()` による疑似乱数
- `hue_shift()` による色相回転
- `smoothstep()` による境界のなだらか化
- `lerp()` による元映像とのブレンド
- `distance()` / `length()` による中心からの距離計算
- `floor()` / `frac()` による格子化、ピクセル化、パーティクル配置

動画編集アプリで再現する際は、これらを以下に置き換えると実装しやすいです。

- 乱数: フレーム番号と座標から決定論的に生成
- ブレンド: 通常合成、加算合成、マスク合成
- ぼかし: 近傍サンプルの平均
- 歪み: UV座標変形

## 7. 動画編集アプリへの移植方針
動画編集アプリで再現する場合、実装は大きく4系統に分けると整理しやすいです。

### 7.1 色変換系
対象例:
- Sepia
- Disco
- Cinema
- Rainy
- Sunset
- Pop High Sat
- Pseudo HDR
- White Balance

実装方針:
- RGBマトリクス
- LUT
- HSL/HSV変換
- トーンカーブ

### 7.2 空間変形系
対象例:
- Vignette
- Old TV
- CRT
- Chromatic Aberration
- Fisheye
- Kaleidoscope
- Mirror
- Split Mirror

実装方針:
- UV座標変形
- 中心基準の距離関数
- マスクによる領域制御

### 7.3 近傍サンプリング系
対象例:
- Shutter 24fps
- Dream
- Sharpen
- Edge Blur
- Background Blur
- Highlight Subject
- Oil Paint
- Watercolor
- Pencil Sketch
- Game Sharp

実装方針:
- 3x3または5x5サンプル
- 平均
- 差分
- ラプラシアン

### 7.4 時間アニメーション系
対象例:
- Disco
- VHS
- Glitch
- Flash
- Strobe
- Fade
- Shake
- Hand Tremor
- Hearts
- Balloons
- Stars
- Snow

実装方針:
- フレーム時間を入力として使用
- 乱数はフレーム番号と座標で決定論的に生成
- パーティクルは格子ベースで疑似生成すると再現しやすい

### 7.5 シーン単位のエフェクトと全体エフェクトの共存
実装上は、シーンごとの設定を全体設定の上に重ねる形が扱いやすいです。

推奨の優先順位:

1. 全体プリセット
2. シーンプリセット
3. 個別コメントの上書き

このとき、各レイヤーは単純な上書きだけでなく、必要に応じて `merge` と `replace` を分けて扱うと便利です。

- `merge`
  - 既存設定を保ちつつ差分だけ反映
- `replace`
  - その項目を完全に置換

## 8. シーン感情プリセット

字幕コメントや1シーンを、感情ラベルで分類して作業を高速化する設計です。

### 8.1 感情ラベル

プルダウンで選べる感情例:

- `neutral`
- `joy`
- `anger`
- `sadness`
- `surprise`
- `fear`
- `embarrassment`
- `teasing`

### 8.2 プリセットの役割

感情ラベルは、見た目と動きを一括で切り替えるための入口です。

たとえば `surprise` なら次のような初期値をまとめられます。

- 吹き出し枠を太くする
- フォントサイズを少し大きくする
- 文字色を強める
- 軽い拡大アニメーションを入れる
- 一瞬だけフラッシュを足す

### 8.3 プリセット適用方式

推奨は「感情ラベル -> プリセット -> 個別微調整」です。

順序:

1. シーンの感情を選ぶ
2. 対応プリセットを適用する
3. 必要なら個別項目を上書きする

この順にしておくと、作業者が毎回細かい設定を作り直さなくて済みます。

## 10. データ構造の推奨

この手の編集では、字幕コメントごとに次のような構造が扱いやすいです。

```json
{
  "scene_id": "scene_001",
  "scope": "scene",
  "emotion": "surprise",
  "preset_id": "emotion_surprise_default",
  "effects": [
    { "effect_type": "flash", "enabled": true },
    { "effect_type": "shake", "enabled": true }
  ],
  "subtitle_style": {
    "font_name": "Meiryo",
    "font_size": 48,
    "bubble_style": "rounded",
    "outline_width": 3
  }
}
```

この形にしておくと、シーン単位でも全体単位でも同じ構造で編集できます。

### 10.1 推奨スキーマの分割

実装側では、次の 3 種類に分けると破綻しにくいです。

#### 全体設定

```json
{
  "project_id": "proj_001",
  "global_effect_preset_id": "global_default",
  "subtitle_global_style_id": "subtitle_standard",
  "default_emotion_preset_id": "emotion_neutral"
}
```

#### シーン設定

```json
{
  "scene_id": "scene_001",
  "start_sec": 12.5,
  "end_sec": 28.0,
  "emotion": "surprise",
  "effect_preset_id": "emotion_surprise_default",
  "subtitle_style_preset_id": "subtitle_bold_speech",
  "notes": "最初の驚きリアクション"
}
```

#### コメント単位設定

```json
{
  "comment_id": "sub_0007",
  "scene_id": "scene_001",
  "text": "え、今の見た?",
  "override": {
    "font_size": 52,
    "bubble_style": "burst",
    "outline_width": 4
  }
}
```

この分割にすると、全体の統一感を保ちながら、特定シーンだけ演出を強める運用がしやすくなります。

### 10.2 エフェクトセットの中身

複数エフェクトを 1 つのまとまりとして扱う場合、1 セットの中に順序を持たせます。

```json
{
  "id": "emotion_surprise_default",
  "type": "effect_group",
  "name": "驚き",
  "blend_mode": "stack",
  "effects": [
    {
      "id": "flash_01",
      "effect_type": "flash",
      "enabled": true,
      "params": {
        "intensity": 0.75,
        "speed": 0.6
      }
    },
    {
      "id": "shake_01",
      "effect_type": "shake",
      "enabled": true,
      "params": {
        "intensity": 0.35,
        "speed": 0.7
      }
    },
    {
      "id": "subtitle_01",
      "effect_type": "subtitle_style",
      "enabled": true,
      "params": {
        "font_name": "Meiryo",
        "font_size": 48,
        "outline_width": 3,
        "bubble_style": "rounded"
      }
    }
  ]
}
```

### 10.3 競合の解決

複数レイヤーが同じ項目を持つ場合は、次の規則にすると扱いやすいです。

1. 全体設定を読む
2. シーン設定を重ねる
3. コメント単位の override を重ねる
4. 最後に手動編集を反映する

同じ項目が複数レイヤーにある場合は、原則として後勝ちにします。

例:

- 全体: フォントサイズ 42
- シーン: フォントサイズ 48
- コメント: フォントサイズ 52

この場合、実際の表示は 52 になります。

### 10.4 プリセットの呼び出し

UI では次のような操作にすると分かりやすいです。

- シーンを選ぶ
- 感情をプルダウンで選ぶ
- プリセットを適用する
- 必要なら個別項目を微調整する
- 複数エフェクトを 1 つのセットとして保存する

おすすめの保存単位は次の 3 つです。

- `emotion_preset`
- `subtitle_style_preset`
- `effect_group_preset`

### 10.5 編集しやすさの優先順位

作業性を優先するなら、UI の見せ方は次がよいです。

- 先に感情を選ばせる
- 次に吹き出し枠と文字スタイルを一括で切り替える
- 最後に細かい数値を触る

これで、毎シーンの個別調整を最低限にできます。

## 11. UI への落とし込み

編集 UI は、少なくとも次の部品に分けると実用的です。

- シーン一覧
- 感情プルダウン
- プリセット一覧
- エフェクトグループ一覧
- 個別エフェクト一覧
- 吹き出し・字幕スタイル編集

### 11.1 シーン一覧

シーン一覧には次を表示します。

- シーン ID
- 開始・終了時間
- 感情ラベル
- 適用中プリセット
- 対象字幕数

### 11.2 感情プルダウン

感情プルダウンは、最低限次を用意します。

- 無感情 / 通常
- 喜び
- 怒り
- 悲しみ
- 驚き
- 恐れ
- 照れ
- からかい

### 11.3 まとめ編集

複数の字幕に同じ演出を当てる場合は、次の2系統を分けると安全です。

- シーン全体への適用
- 1コメントのみへの適用

この分離がないと、あとから修正するときに全体が巻き戻ってしまいます。

## 13. 運用ルール

この構造を運用するときのルールです。

### 13.1 1シーン = 1感情を基本にする

1つのシーンに複数感情を混ぜると管理が崩れやすいので、基本は 1 シーン 1 感情に寄せると扱いやすいです。

### 13.2 例外は comment override に閉じ込める

例外的に強調したい 1 コマだけは、シーン全体を変えずにコメント単位の override で処理します。

### 13.3 複合エフェクトは保存できる単位にする

一度作った複合エフェクトは、次回から 1 つのプリセットとして再利用できるようにします。

### 13.4 デフォルトは壊さない

プリセットを変えても、未指定の項目は全体設定から継承するようにします。

## 14. 実装時のチェックポイント

- 全体設定、シーン設定、コメント設定が別レイヤーになっているか
- `effect_group` を保存できるか
- 1つのシーンに複数エフェクトを束ねられるか
- 感情ラベルのプルダウンからプリセットを切り替えられるか
- 吹き出し枠、フォントサイズ、フォント名を一括適用できるか
- 個別字幕だけ上書きできるか
- 上書き内容が後から追跡できるか

## 15. 他 AI への短い指示文

字幕コメントをシーン単位で管理している前提で、全体エフェクトとシーンエフェクトを分離し、複数エフェクトは `effect_group` として束ねられるようにする。
感情ラベルはプルダウンで選べるようにし、喜び・怒り・驚きなどのプリセットに吹き出し枠、フォントサイズ、フォント名、縁取りをまとめて割り当てる。
全体設定、シーン設定、コメント単位 override の順で重ね、最後に手動編集を反映する。

## 16. 移植時の注意点
### 8.1 この実装は「単一パスの見た目再現」
OBS版は、人物セグメンテーションや深度推定のような高度な解析はしていません。  
そのため `Background Blur` や `Highlight Subject` は、厳密な被写体分離ではなく、画面中央を被写体想定で扱う簡易実装です。

### 8.2 パーティクルは本物の物理シミュレーションではない
Hearts / Balloons / Stars / Snow は、実体のある粒子システムではなく、フラグメントシェーダー内で格子ごとに疑似生成しています。  
動画編集アプリで再現する場合は、2Dエフェクトとして再計算してもよいですし、専用のパーティクルコンポジタに置き換えてもよいです。

### 8.3 速度は「アニメーション速度」であって再生速度ではない
`speed` は映像全体の再生速度を変える値ではありません。  
時間依存の変化量だけを増減します。

### 8.4 エフェクトの一部は元映像の内容に強く依存する
`Skin Tone` や `Text Readability` は入力画面の性質に影響されます。  
再現精度を上げるなら、動画編集アプリ側でもローカル輝度・色差ベースの条件判定を維持した方がよいです。

## 18. 結論
このプラグインの本質は、以下の3点に集約されます。

1. 1つのシェーダーで62種を分岐処理する
2. 共通パラメータは `intensity` / `speed` / `color` / `time`
3. 効果の大半は「色変換」「座標変形」「近傍サンプル」「簡易パーティクル」で再現できる

動画編集アプリ側で再現する時は、まず共通パラメータ体系を合わせ、次にカテゴリごとに処理を分けると移植が容易です。

字幕コメント運用を強めるなら、シーン単位の感情ラベルと全体単位のスタイルプリセットを分ける設計が一番崩れにくいです。

## 20. バリデーション方針

### 20.1 必須検証

- ID が空でないこと
- `start_sec < end_sec` であること
- 参照先の `effect_group_id` が存在すること
- 参照先の `subtitle_style_preset_id` が存在すること

### 20.2 許容検証

- 未知の追加フィールドは基本的に保持する
- ただし、レンダリング前に危険な値は正規化する

### 20.3 正規化

- フォントサイズは内部で丸める
- 色は `#RRGGBB` または `rgba()` に統一する
- 吹き出し形状名は内部列挙値に落とす

## 21. 実装上の注意

複数エフェクトを 1 つのエフェクトとして扱う場合、単純な配列だけにすると後で編集順序が壊れやすいです。

そのため、必ず次を持たせるのが安全です。

- `id`
- `enabled`
- `effect_type`
- `params`
- `blend_mode`

さらに、シーン単位で使う設定と全体共通設定は分けて保存してください。

## 22. 他 AI への短い指示文

字幕のシーン管理を前提に、感情プリセット、吹き出しプリセット、複合エフェクトグループを JSON Schema で検証可能な形に分離する。
全体設定、シーン設定、コメント単位 override の3層を持ち、複数エフェクトは `effect_group` として保存する。
実装時は `id`、`enabled`、`blend_mode`、`params` を必須にし、`emotion` はプルダウン選択可能な列挙値に限定する。

## 23. 画面エフェクトのスタック設計

字幕装飾とは別に、映像全体へかける画面エフェクトは `screen_effect_stacks` として管理する。

### 23.1 保存構造

```json
{
  "screen_effect_stacks": [
    {
      "id": "screen_stack_cinema_soft",
      "name": "シネマ柔らか",
      "description": "落ち着いた色と軽い粒状感の組み合わせ",
      "effects": [
        { "id": "cinema", "intensity": 0.55, "speed": 1, "color": "#ffffff" },
        { "id": "vignette", "intensity": 0.4, "speed": 1, "color": "#000000" }
      ]
    }
  ],
  "screen_effect_targets": {
    "global_stack_ids": ["screen_stack_cinema_soft"],
    "scene_stack_ids": {
      "scene_001": ["screen_stack_vhs"]
    }
  }
}
```

### 23.2 適用順序

1. 全体に適用されたスタックを先に合成する。
2. 対象シーンに紐づいたスタックを後から重ねる。
3. スタック内の `effects` は配列順に評価する。

### 23.3 UI 方針

- スタックは再利用可能な単位として編集する。
- 全体 / 現在シーン / 全シーン の適用ボタンを用意する。
- 字幕装飾の `effect_group` と画面フィルターは別管理にする。
- プレビューでは実装可能な範囲で CSS filter へフォールバックしてもよい。
